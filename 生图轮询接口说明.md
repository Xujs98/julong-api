# 矩龙-API 生图轮询接口说明

本文档说明如何通过矩龙-API 提交生图任务、轮询任务状态以及下载生成结果。

## 1. 功能说明

生图接口支持两种调用方式：

- 同步方式：不传 `async` 或传 `false`，请求会一直等待，直到图片生成完成后返回结果。
- 轮询方式：传入 `"async": true`，接口立即返回任务 ID，调用方再根据任务 ID 查询状态。

轮询方式由矩龙-API 在本地实现，`async` 不会转发给上游渠道。

## 2. 使用前提

管理员需要打开：

> 系统设置 → 运维 → 日志维护 → 记录生图日志

同时可以配置“生图任务刷新频率（秒）”：

- 默认：`15` 秒
- 最小：`5` 秒
- 最大：`3600` 秒

“记录生图日志”区域还可以配置“允许白名单免鉴权读取图片”：

- 默认关闭，关闭时下载本地图片必须携带所属用户的 API Key。
- 支持每行输入一个精确域名或 IP，可输入多条；也支持英文逗号分隔。
- IP 匹配请求的客户端 IP；部署在 Nginx/CDN 后时，必须确保反向代理正确传递并覆盖真实客户端 IP。
- 域名匹配浏览器请求中的 `Origin` 或 `Referer` 主机名，例如 `https://www.example.com/page` 匹配配置项 `www.example.com`。
- 可粘贴完整 HTTP/HTTPS URL，保存时只保留主机名；不支持通配符和 CIDR 网段。
- 输入 `0.0.0.0` 表示所有来源都可以免鉴权读取本地生成图片。
- 白名单只影响图片文件接口；任务状态查询和图片生成始终需要 API Key。

安全提示：`Origin/Referer` 可被非浏览器客户端伪造，敏感场景应优先使用 IP 白名单。任务 ID 仍应视为敏感信息，不要公开分享。

如果“记录生图日志”处于关闭状态：

- `"async": true` 不生效。
- 请求自动按照同步方式执行。
- 不返回任务 ID。
- 不创建生图任务日志。
- 不保存生成图片到本地日志目录。

## 3. 接口地址

生产环境：

```text
https://api.julongkj.top
```

本地开发环境：

```text
http://localhost:5173
```

下文使用环境变量，调用时按实际环境修改：

```bash
BASE_URL="https://api.julongkj.top"
API_KEY="sk-你的API密钥"
```

API Key 必须通过请求头传递：

```http
Authorization: Bearer sk-xxx
```

不要将真实 API Key 写进公开文档、代码仓库、截图或日志。

## 4. 提交生图任务

### 4.1 请求

```http
POST /v1/images/generations
Content-Type: application/json
Authorization: Bearer <API_KEY>
```

### 4.2 完整请求参数

| 参数 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `model` | string | 是 | 无 | 生图模型，例如 `gpt-image-2` |
| `prompt` | string | 是 | 无 | 生图提示词，不能为空 |
| `async` | boolean | 否 | `false` | 设置为 `true` 时启用任务轮询方式 |
| `n` | integer | 否 | `1` | 图片数量，矩龙-API 最大允许 `128`，上游可能有更小限制 |
| `size` | string | 否 | 模型默认 | 图片尺寸，例如 `1024x1024` |
| `quality` | string | 否 | 模型默认 | 图片质量，例如 `standard`、`high`、`hd` |
| `response_format` | string | 否 | 上游默认 | 常见值为 `url` 或 `b64_json` |
| `output_format` | string | 否 | 上游默认 | 输出格式，例如 `png`、`webp` |
| `output_compression` | integer | 否 | 上游默认 | 输出压缩质量，具体范围由模型决定 |
| `background` | string | 否 | 上游默认 | 背景参数，具体值由模型决定 |
| `moderation` | string | 否 | 上游默认 | 内容审核模式 |
| `style` | string/JSON | 否 | 上游默认 | 图片风格参数 |
| `user` | string/JSON | 否 | 无 | 上游终端用户标识 |
| `watermark` | boolean | 否 | 上游默认 | 是否添加水印，由渠道适配器决定 |
| `watermark_enabled` | JSON | 否 | 上游默认 | 智谱 4V 等渠道使用的水印扩展参数 |
| `user_id` | JSON | 否 | 无 | 部分渠道使用的上游用户标识 |
| `stream` | boolean | 否 | `false` | 有效轮询模式下不能同时设置为 `true` |
| `partial_images` | integer | 否 | 上游默认 | 流式局部图片数量；轮询方式不支持流式返回 |
| `image` / `images` | JSON | 否 | 无 | 部分渠道支持的参考图扩展参数 |
| `mask` | JSON | 否 | 无 | 部分渠道支持的遮罩扩展参数 |
| `input_fidelity` | string/JSON | 否 | 上游默认 | 参考图保真参数 |
| `extra_fields` | JSON | 否 | 无 | 渠道适配器支持的扩展参数 |

未声明的扩展字段会根据渠道透传和适配器能力处理，不保证所有上游都支持。

### 4.3 curl 提交示例

```bash
RESPONSE=$(curl -sS -X POST "$BASE_URL/v1/images/generations" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $API_KEY" \
  -d '{
    "model": "gpt-image-2",
    "prompt": "一张极简产品海报",
    "n": 1,
    "size": "1024x1024",
    "quality": "high",
    "async": true
  }')

echo "$RESPONSE" | jq .
```

### 4.4 提交成功响应

接口返回 HTTP `202 Accepted`：

```json
{
  "task_id": "img_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
  "object": "image.generation.task",
  "status": "pending",
  "created_at": 1784422800,
  "poll_url": "/v1/images/generations/img_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx",
  "polling_interval_seconds": 15
}
```

字段说明：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `task_id` | string | 生图任务 ID，后续查询必须使用该值 |
| `object` | string | 固定为 `image.generation.task` |
| `status` | string | 初始状态通常为 `pending` |
| `created_at` | integer | 任务创建时间，Unix 秒级时间戳 |
| `poll_url` | string | 任务查询相对路径 |
| `polling_interval_seconds` | integer | 服务端建议的轮询间隔秒数 |

提取任务信息：

```bash
TASK_ID=$(echo "$RESPONSE" | jq -er '.task_id')
POLL_SECONDS=$(echo "$RESPONSE" | jq -r '.polling_interval_seconds // 15')

echo "任务 ID：$TASK_ID"
echo "轮询间隔：$POLL_SECONDS 秒"
```

如果响应中没有 `task_id` 而是直接出现 `data` 图片结果，请检查“记录生图日志”是否已经打开。

## 5. 查询任务状态

### 5.1 请求

```http
GET /v1/images/generations/:task_id
Authorization: Bearer <API_KEY>
```

查询时必须使用提交任务时所属用户的 API Key。其他用户查询该任务会返回 `404`。

### 5.2 curl 单次查询

```bash
RESULT=$(curl -sS \
  "$BASE_URL/v1/images/generations/$TASK_ID" \
  -H "Authorization: Bearer $API_KEY")

echo "$RESULT" | jq .
```

### 5.3 任务状态

| 状态 | 说明 | 是否继续轮询 |
| --- | --- | --- |
| `pending` | 任务已创建，正在等待执行 | 是 |
| `processing` | 已请求上游，正在生成图片 | 是 |
| `success` | 图片生成成功 | 否 |
| `failed` | 图片生成失败 | 否 |

成功状态的正确拼写是 `success`。

### 5.4 处理中响应示例

```json
{
  "task_id": "img_xxx",
  "object": "image.generation.task",
  "status": "processing",
  "progress": 10,
  "created_at": 1784422800,
  "updated_at": 1784422802,
  "request_id": "20260719031002414087000p8Bq8GBpf4VsoD7E",
  "model": "gpt-image-2",
  "image_count": 0,
  "polling_interval_seconds": 15,
  "data": [],
  "error": null
}
```

### 5.5 成功响应示例

```json
{
  "task_id": "img_xxx",
  "object": "image.generation.task",
  "status": "success",
  "progress": 100,
  "created_at": 1784422800,
  "updated_at": 1784422828,
  "request_id": "20260719031002414087000p8Bq8GBpf4VsoD7E",
  "model": "gpt-image-2",
  "image_count": 1,
  "polling_interval_seconds": 15,
  "data": [
    {
      "url": "/v1/images/generations/img_xxx/images/0"
    }
  ],
  "error": null,
  "response": {
    "created": 1784422828,
    "data": [
      {
        "url": "/v1/images/generations/img_xxx/images/0"
      }
    ]
  }
}
```

### 5.6 失败响应示例

任务执行失败后，查询接口通常仍返回 HTTP `200`，通过 `status=failed` 表示业务失败：

```json
{
  "task_id": "img_xxx",
  "object": "image.generation.task",
  "status": "failed",
  "progress": 100,
  "polling_interval_seconds": 15,
  "data": [],
  "error": {
    "message": "上游返回的错误信息"
  }
}
```

## 6. 自动轮询脚本

下面的脚本使用服务端返回的轮询频率，遇到 `success` 或 `failed` 后自动停止：

```bash
while true; do
  RESULT=$(curl -sS \
    "$BASE_URL/v1/images/generations/$TASK_ID" \
    -H "Authorization: Bearer $API_KEY")

  echo "$RESULT" | jq .

  STATUS=$(echo "$RESULT" | jq -r '.status')

  if [ "$STATUS" = "success" ]; then
    echo "图片生成成功"
    break
  fi

  if [ "$STATUS" = "failed" ]; then
    echo "图片生成失败"
    echo "$RESULT" | jq -r '.error.message // "未知错误"'
    exit 1
  fi

  POLL_SECONDS=$(echo "$RESULT" | jq -r \
    '.polling_interval_seconds // 15')
  sleep "$POLL_SECONDS"
done
```

不要在 `pending` 或 `processing` 状态下进行高频无间隔请求。

## 7. 下载生成图片

成功后从 `data` 中读取图片地址：

```bash
IMAGE_URL=$(echo "$RESULT" | jq -er '.data[0].url')
echo "$IMAGE_URL"
```

### 7.1 下载矩龙-API 本地图片

当图片地址以 `/` 开头时，该图片由矩龙-API 本地接口提供。默认必须携带 API Key：

```bash
curl -L "$BASE_URL$IMAGE_URL" \
  -H "Authorization: Bearer $API_KEY" \
  -o generated-image.png
```

本地图片接口格式：

```http
GET /v1/images/generations/:task_id/images/:index
Authorization: Bearer <API_KEY>
```

如果管理员已经打开图片读取白名单，且当前客户端 IP 或浏览器页面域名命中白名单，可以省略 API Key：

```bash
curl -L "$BASE_URL$IMAGE_URL" -o generated-image.png
```

配置 `0.0.0.0` 后，任何持有图片 URL 的客户端都可使用上述免鉴权命令。该设置不会放开 `GET /v1/images/generations/:task_id` 任务查询，也不会放开 `POST /v1/images/generations` 生图请求。

### 7.2 获取 MinIO 临时直链

如果管理员启用了 MinIO 生图存储，可以为当前 API Key 所属任务的图片生成临时直链：

```bash
curl "$BASE_URL/v1/images/generations/$TASK_ID/images/0/presign?expires_in=900" \
  -H "Authorization: Bearer $API_KEY"
```

成功响应：

```json
{
  "url": "https://media.example.com/julong-media/generated/images/...?X-Amz-Signature=...",
  "expires_in": 900,
  "expires_at": "2026-07-19T17:00:00Z"
}
```

`expires_in` 默认 `3600` 秒，允许 `60-86400` 秒。该接口始终需要 API Key，且只允许访问当前用户自己的任务；非 MinIO 历史图片返回 HTTP 409。

### 7.3 下载上游远程图片

如果 `url` 是以 `http://` 或 `https://` 开头的完整地址，则直接下载：

```bash
curl -L "$IMAGE_URL" -o generated-image.png
```

完整判断脚本：

```bash
case "$IMAGE_URL" in
  http://*|https://*)
    curl -L "$IMAGE_URL" -o generated-image.png
    ;;
  /*)
    curl -L "$BASE_URL$IMAGE_URL" \
      -H "Authorization: Bearer $API_KEY" \
      -o generated-image.png
    ;;
  *)
    echo "无效的图片地址：$IMAGE_URL"
    exit 1
    ;;
esac
```

## 8. 完整一体化示例

该示例完成提交、轮询和下载第一张图片：

```bash
#!/usr/bin/env bash
set -euo pipefail

BASE_URL="https://api.julongkj.top"
API_KEY="sk-你的API密钥"

RESPONSE=$(curl -sS -X POST "$BASE_URL/v1/images/generations" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $API_KEY" \
  -d '{
    "model": "gpt-image-2",
    "prompt": "一张极简产品海报",
    "n": 1,
    "size": "1024x1024",
    "async": true
  }')

echo "$RESPONSE" | jq .

TASK_ID=$(echo "$RESPONSE" | jq -er '.task_id')
POLL_SECONDS=$(echo "$RESPONSE" | jq -r \
  '.polling_interval_seconds // 15')

while true; do
  RESULT=$(curl -sS \
    "$BASE_URL/v1/images/generations/$TASK_ID" \
    -H "Authorization: Bearer $API_KEY")

  STATUS=$(echo "$RESULT" | jq -r '.status')
  echo "任务状态：$STATUS"

  case "$STATUS" in
    success)
      break
      ;;
    failed)
      echo "$RESULT" | jq -r '.error.message // "图片生成失败"'
      exit 1
      ;;
    pending|processing)
      POLL_SECONDS=$(echo "$RESULT" | jq -r \
        '.polling_interval_seconds // 15')
      sleep "$POLL_SECONDS"
      ;;
    *)
      echo "未知任务状态：$STATUS"
      exit 1
      ;;
  esac
done

echo "$RESULT" | jq .
IMAGE_URL=$(echo "$RESULT" | jq -er '.data[0].url')

case "$IMAGE_URL" in
  http://*|https://*)
    curl -L "$IMAGE_URL" -o generated-image.png
    ;;
  /*)
    curl -L "$BASE_URL$IMAGE_URL" \
      -H "Authorization: Bearer $API_KEY" \
      -o generated-image.png
    ;;
esac

echo "图片已保存到 generated-image.png"
```

脚本依赖：

```bash
brew install jq
```

如果系统已经安装 `jq`，不需要重复安装。

## 9. HTTP 状态码和错误处理

| HTTP 状态码 | 场景 | 处理建议 |
| --- | --- | --- |
| `202` | 异步任务创建成功 | 保存 `task_id` 并按建议间隔轮询 |
| `200` | 查询成功，或日志关闭时返回同步图片结果 | 检查响应中的 `status` 或 `data` |
| `400` | JSON、参数、提示词无效，或有效异步模式与流式冲突 | 修正请求体 |
| `401` | API Key 无效、缺失或过期 | 更换有效 API Key |
| `403` | 用户、Token 或 IP 被禁用 | 联系管理员检查账号状态 |
| `404` | 任务不存在或不属于当前 API Key 用户 | 检查任务 ID 和 API Key |
| `503` | 模型没有可用渠道等服务问题 | 检查渠道和模型配置后重试 |

提交接口错误格式示例：

```json
{
  "error": {
    "message": "prompt is required",
    "type": "invalid_request_error",
    "code": "image_generation_task_error"
  }
}
```

## 10. 存储与运行限制

- base64 图片会解码后写入 `image-generation-logs` 目录。
- Docker 默认位于持久化卷中的 `/data/image-generation-logs`。
- 上游返回完整 URL 时，只保存 URL 引用，不主动下载远程图片。
- 图片和日志按“生图日志保留天数”自动清理，`0` 表示永久保留。
- 单个进程最多同时执行 16 个本地异步生图任务，超出的任务会等待执行槽位。
- 异步执行器位于当前进程内。任务执行期间重启后端或容器，任务无法自动恢复，可能停留在 `pending` 或 `processing`。
- 生产环境部署更新前，应尽量等待正在执行的生图任务完成。

## 11. 后台查看

开启日志后，可以在以下位置查看任务：

> 任务日志 → 生图日志

后台支持：

- 查看任务 ID 和实时状态。
- 点击任务 ID 查看脱敏后的任务 JSON。
- 任务未完成时按管理员配置的频率自动刷新。
- 成功后查看和下载图片。
- 下载任务 JSON 数据。
